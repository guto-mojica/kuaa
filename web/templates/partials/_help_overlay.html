{#- Keyboard help overlay (Phase 7 / Task 28). Rendered once into base.html at page load (hidden); ``mojica.js`` toggles ``[hidden]`` on the outer ``.kh-back`` div in response to the ``?`` keypress. The DOM here is purely server-rendered (no HTMX) so the open path can render its first frame instantly without any network round trip — the user mashed `?` to find a shortcut, and making them wait for an XHR before reading the legend would defeat the point. Class names follow web/static/css/polish.css verbatim: * ``.kh-back`` — backdrop (fixed, full-viewport, blurred) * ``.kh-panel`` — centred panel (720px max) * ``.kh-head``, ``.kh-head h2``, ``.kh-head h2 .pip`` (the "?" badge), ``.kh-head .close`` (the dismiss button) * ``.kh-body`` — 2-column grid container * ``.kh-group`` — one column of shortcuts under an h3 heading * ``.kh-row`` — single shortcut row (description + keys) * ``.kh-row .desc``, ``.kh-row .keys``, ``.kh-row .keys .k``, ``.kh-row .keys .sep`` Mutual exclusion with the command palette is enforced on the JS side (see ``window.Help`` and the ⌘K block in mojica.js): opening one closes the other, so a tap of ``?`` while the palette is open swaps the two surfaces cleanly. Shortcut taxonomy mirrors the prototype's `polish.jsx` HelpOverlay component (Navigation · Universal · Annotate review · Eval grading · Cenas + Buscar). The Eval row is wired even though Task 30 ships the eval surface itself — the legend documents what those shortcuts will be once the page lands, so users discover the system through the help overlay rather than after stumbling onto /eval blind. -#} {% from "partials/_icons.html" import icon %}
