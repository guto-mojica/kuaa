Go deeper · Domain packs

Configuration over code.

KUAA isn't hardcoded for one film archive. A Domain Pack is a YAML file that defines the domain-specific vocabulary — prompts, metadata fields, taxonomy, filters, and export mapping — around a processing engine that stays exactly the same: scene detection, keyframe extraction, embeddings, and the search/hybrid/rhymes/rerank stack never change.

The domain pack concept

One engine, two vocabularies today.

The pipeline still detects scenes, generates keyframe metadata, creates embeddings, and writes JSON files exactly as before. A domain pack defines the language around that engine: which prompts the scene describer sends, which metadata fields get parsed out, which taxonomy values populate filters, and how the final record is shaped for export.

Selection is config-driven: domain.pack: archive loads config/domains/archive.yaml by default, or an explicit domain.path loads any pack file directly. Both Moondream scene-describer backends read the selected pack and iterate its prompt_templates — the UI renders the shared metadata fields, while extra domain responses stay available under _raw_responses and through export_mapping.

Side by side

Archive vs. media_broadcast.

Two packs ship today: archive (the default, film-cataloging behavior) and media_broadcast (b-roll review, clip discovery, licensing triage). What stays identical across both is scene detection, keyframe extraction, visual embeddings, the search/hybrid/rhymes/rerank stack, and the JSON/CSV export machinery — only prompts, fields, taxonomy, filters, and export mapping move into the pack YAML.

How it works

What a pack file has to define.

Required top-level fields: id (a stable machine id such as archive), label (human-facing name), metadata_fields (an ordered list of field definitions), prompt_templates (prompt key → prompt text and token budget), and export_mapping (export field name → metadata path, using dot notation, so a pack can define a structured export shape independent of the export endpoint's own code).

Optional fields: description, taxonomy, filters, sample_outputs, evaluation.

Adding a new domain means writing a YAML file, not touching pipeline code: define the metadata fields and prompt templates, add taxonomy terms and filters, add an export mapping, add a small evaluation query file, and run the domain-pack test suite. tests/test_domain_packs.py proves both existing packs load and map exports without any pipeline edits.
