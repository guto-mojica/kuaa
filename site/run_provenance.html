Go deeper · Run provenance

Every catalog record traces back to a run.

For technical reviewers, reproducibility matters as much as accuracy. Every pipeline execution writes a machine-readable receipt — run_manifest.json — that answers: which input, which config, and which model versions produced this specific catalog?

The run manifest

What gets recorded, every run.

A shared manifest helper writes the same record after success, error, a blocked downstream step, or user cancellation — not only on a clean run. Manifest writing is best-effort: a write failure is logged but never overrides the real pipeline result.

  • input path, existence, size, and modified time
  • a config SHA-256 hash plus a JSON-compatible config snapshot
  • the selected domain pack's id, label, and path
  • configured model backends and model revision hints
  • per-step states, durations, errors, and output paths
  • expected output artifact paths and whether each exists
  • run start/end timestamps and terminal status

Example (illustrative shape)

{
  "input": {
    "path": "/data/raw/jeca_tatu_1959.mp4",
    "exists": true,
    "size_bytes": 812345678
  },
  "config_hash": "sha256:a9b3c4d1f...",
  "domain": { "id": "archive", "label": "Film archive" },
  "model_backends": {
    "image_embedder": "siglip_multilingual",
    "scene_describer": "moondream_transformers",
    "object_detector": "yolov8"
  },
  "steps": {
    "frame_extraction": { "status": "done", "duration_s": 42.1 },
    "llm_description": { "status": "done", "duration_s": 5123.4 }
  },
  "terminal_status": "done"
}

Operational impact

Why this matters beyond auditing.

Traceable, not just logged

Because every field is structured and every run writes one, a config change or model swap that shifts retrieval quality can be traced back to the exact run that caused it — not reconstructed from memory or scattered log lines.

Full field-by-field detail lives in docs/OPERATIONS.md.
