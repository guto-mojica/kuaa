Go deeper · Architecture

From a raw video file to an indexed catalog.

KUAA has two surfaces: a batch pipeline that turns a video into keyframes, metadata, embeddings, and descriptions, and a FastAPI + HTMX web interface for search, browsing, annotation, visual-rhyme discovery, eval grading, and exports. Both sit on top of a registry-backed multi-film library under data/library/<slug>/....

High-level flow

One video, six stages, one artifact tree.

Each stage reads the previous stage's output and writes its own artifacts under the film's directory. Downstream steps are blocked — not silently skipped — when a required input is missing or an in-run prerequisite failed.

01

FFprobe

Inspects the source file → video_properties.json

02

FFmpeg extraction

Decodes and samples frames → frames/sample/*.jpg

03

PySceneDetect

Finds cut boundaries, stores keyframes → keyframes_metadata.json

04

Visual analysis

Face detection, object detection, environment heuristic → visual_analysis.json

05

Embeddings

SigLIP2 keyframe vectors → keyframe_embeddings.npy

06

LLM descriptions

Moondream 2 writes scene text and tags → scene_descriptions.json

STEP_ORDER: frame_extraction → scene_detection → visual_analysis → embeddings → llm_description

Component deep dive

Pipeline modules and the web application.

Processing (src/kuaa/)

pipeline.py orchestrates the steps; data_prep.py, scene_detector.py, visual_analyzer.py, and embeddings.py implement each stage. library/ holds the registry and per-film path context; search/, rhymes/, and eval/ implement retrieval, cross-film rhymes, and evaluation metrics.

Web app (api/)

Each tab is a route module: search.py, scenes.py, annotate*.py, processing.py, rimas.py, eval.py, export.py. Route handlers call into api/services/, which in turn call src/kuaa/ — the AI core has no import path back into api/.

The UI is server-rendered: HTMX swaps tab partials into the page, and server-sent events stream pipeline progress. Direct routes such as /search and HTMX tab routes share the same context builders, so a bookmarked URL and an in-app tab switch render identically.

Architecture philosophy

Model roles are typed Protocols, not hard imports.

The pipeline asks a registry for a model role — it never imports a concrete model class directly. That makes it possible to add an ONNX, quantized, fine-tuned, or domain-specific backend later without rewriting the orchestration layer.

Five typed Protocols

ImageEmbedder, FaceDetector, ObjectDetector, SceneDescriber, and EnvironmentClassifier are defined in src/kuaa/models/base.py. Concrete backends are selected by src/kuaa/models/registry.py from the models: section of config.

Rationale for the Protocol-based design is documented in docs/PROTOCOL_OPTION.md.

Current constraints

What this architecture is, and isn't, today.

  • Semantic search is a NumPy dot product over local embeddings, not a vector database — appropriate for small/medium demo collections.
  • Cross-film aggregate retrieval scans local per-film indices; it is not a distributed vector service.
  • The cross-encoder reranker is off by default — its effect is unmeasured on short captions, and its text-only design is suspect (see docs/RERANKER_DECISION.md).
  • Run manifests still write to the configured flat metadata directory rather than the active film's per-film metadata directory.
  • The environment classifier is a heuristic, not a trained archive-specific model.
  • The default object detector, Ultralytics YOLOv8, carries AGPL licensing implications — see docs/MODEL_INVENTORY.md.
  • Model downloads happen on first use unless weights are already cached.
  • Full video processing can be slow on CPU, especially scene description.
